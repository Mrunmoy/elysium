Section 12 SPI / I2C / DMA Drivers

"Every peripheral is a state machine waiting to be understood."
— Embedded systems proverb

Phase 12 adds three register-level HAL modules for the STM32F2/F4 bus peripherals. Each module provides both polled (blocking) and interrupt-driven (async) transfer APIs. The Zynq-7000 targets receive no-op stubs since the Cadence SPI/I2C IP on the Zynq PS has a different register layout.

DMA Controller

The STM32 DMA subsystem has two controllers (DMA1 at 0x40026000, DMA2 at 0x40026400), each with 8 streams. Each stream can be assigned one of 8 channels, connecting it to a specific peripheral request line. Per-stream registers are at base + 0x10 + stream * 0x18.

DMA Transfer Flow dmaInit(config) | Configure stream: channel, direction, sizes, increments, priority | Do not enable yet (EN=0) v dmaStart(controller, stream, periphAddr, memAddr, count, callback, arg) | Store callback in s_state[controller][stream] | Write NDTR, PAR, M0AR | Clear status flags | Set EN=1 in SxCR v Hardware Transfer | DMA moves data autonomously (periph<->memory) | CPU is free to do other work v ISR (transfer complete or error) | Read LISR/HISR for flags | Clear flags via LIFCR/HIFCR | Disable stream (EN=0) | Invoke callback(arg, flags) v dmaStop() -- manual abort if needed
FunctionDescription
dmaInit(config)Configure stream parameters (channel, direction, sizes, priority)
dmaStart(ctrl, stream, ...)Start a transfer with callback notification
dmaStop(ctrl, stream)Abort an in-progress transfer
dmaIsBusy(ctrl, stream)Check if stream is currently transferring
dmaRemaining(ctrl, stream)Read NDTR for remaining transfer count

SPI (Serial Peripheral Interface)

Three SPI instances: SPI1 on APB2 (0x40013000), SPI2/SPI3 on APB1 (0x40003800, 0x40003C00). Configured as master with software NSS management. Four clock modes (CPOL/CPHA combinations), 8 baud rate prescaler settings (Div2 through Div256).

SPI Polled Transfer (per byte) 1. Wait TXE=1 (transmit buffer empty) 2. Write byte to DR (or 0xFF if tx buffer is null) 3. Wait RXNE=1 (receive buffer not empty) 4. Read byte from DR (or discard if rx buffer is null) 5. Repeat for each byte

The async transfer uses per-instance state (s_spiState[3]) with interrupt-driven byte pumping. The ISR processes RXNE first (read received byte), then TXE (write next byte). When all bytes are received, interrupts are disabled and the completion callback is invoked.

FunctionDescription
spiInit(config)Configure SPI: mode, prescaler, data size, bit order, master/slave
spiTransfer(id, tx, rx, len)Polled full-duplex transfer (tx or rx may be null)
spiTransferByte(id, byte)Convenience: send one byte, return received byte
spiTransferAsync(id, tx, rx, len, cb, arg)Interrupt-driven transfer with completion callback

I2C (Inter-Integrated Circuit)

Three I2C instances, all on APB1: I2C1 (0x40005400), I2C2 (0x40005800), I2C3 (0x40005C00). Standard mode (100 kHz) and Fast mode (400 kHz). 7-bit addressing. The STM32 I2C peripheral requires careful sequencing for ADDR flag clearing, byte-count-dependent ACK/NACK timing, and STOP generation.

I2C Write Sequence START -> wait SB -> addr<<1|0 -> wait ADDR -> clear ADDR (read SR1+SR2) -> for each byte: wait TXE -> write DR -> wait BTF -> STOP I2C Read Sequence (N > 2 bytes) START -> wait SB -> addr<<1|1 -> wait ADDR -> clear ADDR -> for each byte: wait RXNE -> read DR -> before last 2 bytes: clear ACK, set STOP -> read final 2 bytes I2C WriteRead (register read pattern) Write phase (no STOP) -> repeated START -> Read phase -> STOP
Key Decision: The readDiscard() helper reads a volatile register into a volatile local variable to guarantee the load instruction is generated. This avoids the ARM GCC -Werror=conversion-to-void warning that occurs when casting a volatile read to (void). The pattern is used for I2C ADDR flag clearing (must read SR1 then SR2) and is documented in the project gotchas file.
FunctionDescription
i2cInit(config)Configure I2C: speed, analog/digital filters, CCR/TRISE timing
i2cWrite(id, addr, data, len)Polled write to slave (returns I2cError)
i2cRead(id, addr, data, len)Polled read from slave (returns I2cError)
i2cWriteRead(id, addr, tx, txLen, rx, rxLen)Write then read with repeated START
i2cWriteAsync(id, addr, data, len, cb, arg)Interrupt-driven write with callback
i2cReadAsync(id, addr, data, len, cb, arg)Interrupt-driven read with callback
