Chapter 15: The Dice Roll

Hardware random number generation on the STM32F4

True randomness from silicon

Software pseudo-random number generators are fine for shuffling playlists, but when an embedded system needs unpredictable values -- cryptographic keys, nonces, session tokens -- only hardware entropy will do. The STM32F4 contains a True Random Number Generator (TRNG) that derives entropy from analog noise sources inside the chip. Three registers, one peripheral clock bit, and a few lines of code: this was going to be the simplest driver yet.

Three registers and a gotcha

The RNG peripheral lives at 0x50060800 on the AHB2 bus. Its entire register set is just three words:

OffsetRegisterPurpose
0x00RNG_CRControl: bit 2 (RNGEN) enables the generator
0x04RNG_SRStatus: DRDY, clock error (CECS), seed error (SECS)
0x08RNG_DRData: 32-bit random value (read clears DRDY)

The clock source is PLL48CLK -- the same 48 MHz output from the PLL Q divider that feeds the USB peripheral. On the F407, PLL Q = 7 gives exactly 48 MHz from the 336 MHz VCO. The RCC gate is bit 6 of AHB2ENR.

RM0090, section 24.3.4: "The first random number generated after setting the RNGEN bit should not be used." The analog noise sources need one generation cycle to seed properly. Skipping this value is mandatory.

The Mbed-OS TRNG driver confirmed this: their trng_init calls HAL_RNG_GenerateRandomNumber once and discards the result before returning. Our rngInit() does the same -- waits for DRDY, reads and discards the data register, then returns.

Error detection

The RNG can detect two fault conditions: CECS (clock error) means the PLL48CLK frequency is wrong or missing, and SECS (seed error) means the noise source produced an anomalous sequence. Both are checked inside the waitDrdy() polling loop. If either flag is set, rngRead() returns msos::error::kIo without writing to the output buffer. A timeout of one million iterations prevents infinite hangs if the peripheral is broken.

The serial capture lesson

The hardware test passed on the first flash -- the interesting part was seeing it pass. The rng-demo app prints ten random values and a summary, then enters WFI. The entire output takes a few milliseconds. When we opened the serial port after resetting the CPU, we captured zero bytes every time. The output had already been sent and lost.

The fix: open the serial port before triggering the reset. Start cat /dev/ttyACM0 in the background, then issue openocd reset run. The UART data arrives into an already-listening file descriptor and nothing is lost.

Rule of thumb: For bare-metal apps that print and halt, always open the serial listener before resetting the target. The CPU is faster than your shell.

Sample output from the TRNG:

0x45E821B9 0xBA1CAD75 0xBC840C9F 0x56F80603 0xA7FF3233
0xE6EC336F 0xA6F99A27 0x3B06E583 0x4F18F610 0xA2D45547

Ten unique, high-entropy, non-zero values -- genuine randomness from analog noise, not an LFSR or deterministic sequence. The smallest driver in the HAL, but the one that makes cryptographic operations possible.
