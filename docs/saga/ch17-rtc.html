Chapter 15: Keeping Time

Phase 21: RTC -- real-time clock driver for the STM32F4 backup domain. Calendar time, BCD encoding, alarm interrupts, and the quiet hum of a 32 kHz oscillator.

The Backup Domain

Most peripherals on the STM32 live and die with VDD. Power off the chip and everything vanishes -- GPIO states, timer counts, UART FIFOs. The RTC is different. It sits in the backup domain, a small island of silicon powered by VBAT. Wire a coin cell to the VBAT pin and the clock keeps ticking even when the rest of the chip is dark.

The backup domain comes with its own set of rules. Before you can touch any register in this domain, you must unlock the gate:

  1. Enable the PWR peripheral clock (APB1ENR bit 28)
  2. Set the DBP bit in PWR_CR to allow backup domain writes
  3. Enable the clock source (LSI or LSE) and select it in BDCR

Skip any of these steps and the RTC registers silently ignore your writes. No fault, no error flag -- just silence.

BCD: An Old Encoding for an Old Problem

The RTC stores time and date in Binary-Coded Decimal. Each decimal digit occupies four bits: the value 23 is stored as 0x23, not 0x17. This is a design choice from the days when displaying digits on seven-segment displays was the primary use case -- BCD maps directly to display hardware without division.

Two small helpers handle the conversion:

std::uint8_t toBcd(std::uint8_t val)
{
    return ((val / 10) << 4) | (val % 10);
}

std::uint8_t fromBcd(std::uint8_t bcd)
{
    return (bcd >> 4) * 10 + (bcd & 0x0F);
}

The time register packs hours, minutes, and seconds into a single 32-bit word. The date register does the same with year, month, day, and weekday. Reading and writing these registers requires entering and exiting init mode -- a two-step handshake where you set the INIT bit, wait for INITF, write your values, then clear INIT to let the clock resume.

Write Protection

ST added a two-key unlock sequence to prevent accidental writes to RTC registers. You must write 0xCA then 0x53 to the Write Protection Register (WPR). Any wrong key immediately re-enables protection. After your writes, send 0xFF to lock it again.

This means every RTC operation follows the same ritual: unlock, do work, lock. The pattern shows up in rtcSetTime, rtcSetDate, rtcSetAlarm, and rtcCancelAlarm.

The LSI Clock

We chose the LSI (Low Speed Internal) oscillator as the default clock source. It is an on-chip RC oscillator running at roughly 32 kHz -- not exactly 32,768 Hz like a crystal would provide, but close enough for many applications and it requires no external components.

The prescaler math adjusts for this:

  • LSI (~32 kHz): PREDIV_A=127, PREDIV_S=249 gives 32000 / (128 * 250) = 1 Hz
  • LSE (32.768 kHz): PREDIV_A=127, PREDIV_S=255 gives 32768 / (128 * 256) = 1 Hz

The LSI frequency varies per chip (the datasheet says 17-47 kHz typical range, with 32 kHz nominal). For precise timekeeping, an external 32.768 kHz crystal on the LSE pins is the right choice. For our tests, LSI drift of a few percent over minutes is acceptable.

Alarm A: Waking Up on Schedule

The RTC provides two alarm units (A and B). We implemented Alarm A with a flexible mask system. Each field (hours, minutes, seconds, date) has a mask bit:

  • Mask = 1: ignore this field (wildcard)
  • Mask = 0: field must match for the alarm to fire

All masks set to 1 means the alarm fires every second. Only seconds unmasked with seconds=30 means the alarm fires once per minute when the seconds reach 30. This gives applications fine-grained control without multiple alarm configurations.

The alarm triggers through EXTI line 17, which requires its own configuration: unmask the interrupt, set rising-edge trigger, and enable NVIC IRQ 41. The ISR checks the ALRAF flag, clears it, clears the EXTI pending bit, and calls the user-registered callback.

The API

Eight functions, no state objects, no dynamic allocation:

void rtcInit(RtcClockSource clockSource);
void rtcSetTime(const RtcTime &time);
void rtcGetTime(RtcTime &time);
void rtcSetDate(const RtcDate &date);
void rtcGetDate(RtcDate &date);
void rtcSetAlarm(const RtcAlarmConfig &config, RtcAlarmCallbackFn callback, void *arg);
void rtcCancelAlarm();
bool rtcIsReady();

The structs use plain uint8_t fields. RtcTime holds hours/minutes/seconds. RtcDate holds year (0-99, offset from 2000), month, day, and weekday (1=Monday). RtcAlarmConfig adds mask booleans for each field.

Testing: 18 Host Tests + 5 Hardware Tests

The host tests follow the established link-time mock substitution pattern. MockRtc.cpp records all calls into global vectors in MockRegisters.h. The tests verify that the API correctly records init calls, set/get time round-trips, alarm configuration, and cancel counting.

The hardware test runs five cases on the STM32F407:

  1. Init and ready: rtcInit(Lsi), verify rtcIsReady() returns true
  2. Set/get time: Set 14:30:00, read back, verify BCD round-trip
  3. Set/get date: Set 2026-03-06 (Friday), read back, verify all fields
  4. Time advances: Set 12:00:00, wait 3 seconds, read 12:00:03
  5. Alarm fires: Set alarm matching seconds=3, verify callback fires within 5 seconds

All five pass on the first run after fixing one subtle issue: the microsecond delay function (timerDelayUs) requires TIM7 to be fully initialized with timerInit(), not just clock-enabled with rccEnableTimerClock(). Without the prescaler and period configuration, the delay returns instantly and time-dependent tests appear to fail because they read the RTC before it has a chance to tick.

What Comes Next

With a working RTC, the OS can now:

  • Timestamp log messages and crash dumps
  • Wake from low-power modes at scheduled times
  • Provide wall-clock time to user applications
  • Schedule periodic tasks aligned to calendar boundaries

The Alarm B unit and the wakeup timer are left for future work -- they follow identical patterns and can be added when needed.
