Chapter 14: Counting Time

Phase 19 -- Hardware timer abstraction for STM32F4

The next peripheral

With SPI, I2C, DMA, and UART all validated across physical wires, the HAL was starting to feel substantial. But one category of peripheral had been conspicuously absent: hardware timers. The STM32F407 has fourteen of them, spanning basic counters (TIM6, TIM7), general-purpose timers with PWM (TIM2-TIM5), and advanced timers with complementary outputs (TIM1, TIM8). Every embedded system eventually needs precise timing -- whether for periodic interrupts, PWM outputs, or microsecond-resolution delays.

Scoping the work

Fourteen timers is a lot of silicon to abstract. The design decision was to start with the six most commonly used: TIM2 through TIM7. TIM2 and TIM5 have 32-bit counters; TIM3, TIM4, TIM6, and TIM7 are 16-bit. TIM6 and TIM7 are "basic" timers with no output channels -- perfect for periodic interrupts and delay generation. TIM2-TIM5 are general-purpose with four capture/compare channels each, ideal for PWM output.

The advanced timers (TIM1, TIM8) with their complementary outputs, break inputs, and dead-time generators were deliberately left for a future phase. Same for input capture, encoder interface, and DMA-driven timer operations.

Clock arithmetic

All six timers sit on the APB1 bus. On the STM32F407, APB1 runs at 42 MHz with a prescaler of /4, but the timer clock gets doubled when the APB prescaler is not 1. So the timer input clock is 84 MHz. The prescaler register (PSC) divides this by (PSC + 1), and the auto-reload register (ARR) determines the overflow period.

Timer frequency formula: f_overflow = timer_clock / ((PSC + 1) * (ARR + 1))
For 1 kHz on F407: PSC = 83 (84 MHz / 84 = 1 MHz tick), ARR = 999 (1 MHz / 1000 = 1 kHz).

The API

The timer HAL follows the same pattern as every other driver: a public header in hal/inc/hal/Timer.h, register-level implementation in hal/src/stm32f4/Timer.cpp, a Zynq stub, and link-time mock for host tests.

The API covers five areas:

  • Basic timer: timerInit, timerStart (with ISR callback), timerStop
  • Counter access: timerGetCount, timerSetCount
  • Runtime update: timerSetPeriod, timerSetPrescaler
  • PWM output: timerPwmInit, timerPwmStart, timerPwmStop, timerPwmSetDuty
  • Utility: timerDelayUs (blocking microsecond delay via TIM7), rccEnableTimerClock, rccDisableTimerClock

Register-level implementation

The STM32F4 implementation directly manipulates CR1, DIER, SR, EGR, CCMR, CCER, CNT, PSC, ARR, and CCRx registers. Each timer has its own ISR handler (TIM2_IRQHandler through TIM7_IRQHandler) that checks the update interrupt flag (UIF), clears it, and invokes the registered callback.

PWM configuration uses output compare mode 1 (OCxM = 110 in CCMR) with preload enabled, and the polarity is controlled through the CCER register. The channel mapping handles all four channels across the two CCMR registers.

Testing

32 host tests verify the mock layer: init records config, start records callbacks, stop clears state, PWM operations record channel actions, counter get/set works, and invalid timer IDs are handled gracefully.

The hardware test runs four scenarios on the STM32F407:

  1. TIM6 periodic interrupt: 1 kHz overflow, count callbacks for ~1 second. Got 1085 callbacks (the busy-loop reference overshoots slightly, not a timer error).
  2. TIM7 microsecond delay: 10 ms delay measured against TIM6 reference counter. Exactly 10000 ticks -- perfect accuracy at PSC = 83.
  3. TIM3 PWM output: 1 kHz PWM on PB4 (AF2), init/start/setDuty/stop without faults.
  4. TIM5 start/stop: verify the 32-bit counter freezes after timerStop. Running count was 65, stopped count stayed at 66.
Hardware result: 5/5 PASS on STM32F407ZGT6 (Board 1, J-Link).

Logic analyzer verification

To go beyond "it didn't crash," we connected a Saleae logic analyzer to PB4 and captured the PWM output at 4 MHz sample rate. A dedicated timer-logic app cycled through four frequency/duty combinations:

The 0.25 us jitter equals exactly one sample period at 4 MHz -- meaning the actual hardware jitter is below measurement resolution. The timer peripheral generates cycle-accurate waveforms from the 84 MHz APB1 timer clock.

What it enables

With hardware timers in place, the OS now has the building blocks for:

  • Kernel software timer wheels (callback scheduling without ISR-per-timer overhead)
  • PWM-driven actuators (motor control, LED dimming, servo positioning)
  • Precise microsecond-resolution delays for protocol timing
  • Frequency measurement and pulse counting (once input capture is added)
