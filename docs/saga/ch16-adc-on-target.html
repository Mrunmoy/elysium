Chapter 16 Listening to Silicon

"The ADC is where software stops guessing and starts measuring."
— Lab notebook, Phase 20

Phase 20 focused on one thing: the STM32F4 ADC path from register write to real conversion result. The new HAL introduced adcInit() and adcRead() for single-shot, polled conversions with global status-code returns. Host tests locked down argument validation, status mapping, and call behavior first. Then the real work started on hardware.

To avoid extra wiring, the first on-target validation used internal ADC channels:

  • Channel 17 — VREFINT
  • Channel 16 — temperature sensor

Those channels removed board-to-board dependencies and answered the core question directly: can the driver configure ADC1, trigger SWSTART, observe EOC, and return stable non-zero samples?

The Debug Detour

Early runs looked broken: no UART output appeared on the expected console port, despite successful flashing and a running core. The fix was procedural: start serial capture first, then reset-and-run the target through J-Link. With that sequence, console logs appeared reliably and the machine-parseable ADC case lines became visible.

On-Target Result

The hardware run on STM32F407 with J-Link reported: 

MSOS_CASE:adc:vrefint-read:PASS
MSOS_CASE:adc:temp-read:PASS
MSOS_CASE:adc:invalid-channel:PASS
MSOS_SUMMARY:adc:pass=3:total=3:result=PASS

That gave us an end-to-end validation chain for Phase 20: design → host TDD → cross-build → on-target execution.

Status after Phase 20: ADC single-shot driver implemented for STM32F4, host-tested, and verified on hardware through internal-channel sampling. Total test count: 477 C++ host tests, 145 Python tests.
